# 🏆 Day 6 Exercises: Kubernetes Networking Challenges

Complete these challenges to solidify your understanding of Kubernetes networking, service discovery, and routing mechanics.

## 🎯 Challenge 1: Resolve a Broken Backend Selector
You are handed a running service that clients are unable to reach. The deployment is running fine, but any requests to the service result in connection timeouts.

### Task
1. Inspect the following broken manifest configuration:
   ```yaml
   apiVersion: apps/v1
   kind: Deployment
   metadata:
     name: secure-auth-api
     labels:
       app.kubernetes.io/name: secure-auth-api
   spec:
     replicas: 2
     selector:
       matchLabels:
         app: auth-api-pod
     template:
       metadata:
         labels:
           app: auth-api-pod
       spec:
         containers:
         - name: api
           image: paulbouwer/hello-kubernetes:1.10
           ports:
           - containerPort: 8080
   ---
   apiVersion: v1
   kind: Service
   metadata:
     name: auth-service
   spec:
     type: ClusterIP
     selector:
       app.kubernetes.io/name: secure-auth-api
     ports:
     - port: 80
       targetPort: 8080
   ```
2. Identify the label mismatch between the Service and the Deployment Pod template.
3. Write the corrected Service manifest so that traffic routes to the pods.
4. **Validation Command**: Write down the `kubectl` command to view the list of endpoints mapped to `auth-service` to confirm it is no longer empty.

## 🎯 Challenge 2: Enable Topology-Aware Routing
In a multi-zone production cluster, you want to ensure traffic generated by a Pod in availability zone `us-east-1a` is routed **exclusively** to the database helper Pod in that same availability zone, unless no healthy instances exist there.

### Task
1. Write a Service manifest named `local-db-helper` routing to pods with labels `app=db-helper`.
2. Add the correct annotation to activate **Topology-Aware Routing**.
3. **Question**: Under what circumstances will Kubernetes ignore the topology hint and route traffic cross-zone anyway? Explain the safety safeguards built into the topology controller.

---

## 🎯 Challenge 3: Verify DNS Search Path Performance
You suspect that your application is overloading CoreDNS by performing multiple search-suffix queries for every external API call.

### Task
1. Deploy the `dns-debug` pod from the `manifests/` folder.
2. Exec into the pod and use `tcpdump` (already installed in `netshoot`) to capture outgoing DNS packets on port 53.
3. In a separate terminal, trigger a curl request to `google.com` and inspect the captured traffic.
4. **Question**: How many queries did your pod send to CoreDNS before receiving the final IP? How can you modify the curl command or the pod configuration to reduce this to a single query?

## 🔑 Answers and Verification Hints

### Challenge 1 Answer
* **Mismatch**: The Service selector looks for `app.kubernetes.io/name: secure-auth-api`, but the Pod template labels are `app: auth-api-pod`.
* **Fix**: Change the Service selector to match the Pod label:
  ```yaml
  spec:
    selector:
      app: auth-api-pod
  ```
* **Validation**: Run `kubectl get endpoints auth-service` or `kubectl get endpointslices -l kubernetes.io/service-name=auth-service`.

### Challenge 2 Answer
* **Service Annotation**:
  ```yaml
  metadata:
    annotations:
      service.kubernetes.io/topology-mode: Auto
  ```
* **Safeguard Logic**: The controller evaluates the ratio of endpoints in each zone vs. the CPU capacity of the nodes in that zone. If the distribution is uneven (e.g., zone A has 90% of the traffic but only 10% of the endpoints), the controller disables topology-routing to prevent overloading local endpoints, falling back to random cluster-wide load balancing.

### Challenge 3 Answer
* **DNS Capture Command**:
  ```bash
  # Inside dns-debug pod
  tcpdump -n -i any port 53
  ```
* **Resolution**: If calling `curl google.com`, the resolver appends search paths because `google.com` contains only one dot (less than the default `ndots:5` threshold). This causes 5 sequential lookups.
* **Optimization**: Call `curl google.com.` (with a trailing dot) to perform an absolute lookup, immediately resolving the address in 1 query.
